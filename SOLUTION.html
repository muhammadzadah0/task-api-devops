1 Containerization

Dockerfile — Multi-stage Build

Stage 1: Builder — Compile wheels dari requirements.txt

FROM python:3.11-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip wheel --no-cache-dir --no-deps --wheel-dir /app/wheels -r requirements.txt

Stage 2: Runtime — Hanya copy wheels + kode, tanpa build tools

FROM python:3.11-slim
WORKDIR /app
RUN addgroup --system appuser && adduser --system --ingroup appuser appuser
COPY --from=builder /app/wheels /wheels
RUN pip install --no-cache-dir /wheels/*
COPY app.py .
RUN chown -R appuser:appuser /app
USER appuser
EXPOSE 8000
HEALTHCHECK --interval=10s --timeout=5s --start-period=5s --retries=3 \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "--workers", "2", "--threads", "2", "app:app"]

Multi-stage build

Stage 1 compile wheels. Stage 2 hanya copy wheels. Build tools (gcc, libpq-dev) tidak masuk image akhir. Image ~130MB vs ~1GB.

Base image

python:3.11-slim — Debian minimal, cukup untuk Python runtime, tanpa package tidak perlu.

Non-root user

appuser. Jika container terkompromi, attacker tidak punya akses root ke host.

HEALTHCHECK

Cek /health tiap 10 detik. 3 kali gagal = container restart otomatis.

No secrets

Tidak ada password/token di image. Semua via environment variable saat runtime.

Gunicorn

Production WSGI — multi-worker, multi-thread, graceful shutdown. Bukan Flask dev server.

docker-compose.yml

ServiceImagePortHealthcheckRestart
dbpostgres:15-alpine5432pg_isreadyunless-stopped
appBuild local8000curl /healthunless-stopped
  • Volume — data PostgreSQL persist walau container restart/hapus
  • Environment variable — via .env (gitignored), template .env.example
  • Startup ordering — depends_on + condition: service_healthy. App jalan setelah DB siap
  • docker compose up langsung jalan tanpa langkah manual

3 Infrastructure as Code

Pilihan: Terraform + Docker provider (kreuzwerker/docker)

Alasan: Tanpa kredensial cloud. validate, plan, apply langsung di lokal. Portabel, idempotent, modular.

FileFungsi
terraform/variables.tfInput: app_port, db_password, db_name, db_user, image_tag
terraform/outputs.tfOutput: app_url, grafana_url, prometheus_url
terraform/main.tfResource: network, volumes, PostgreSQL, app (build dari Dockerfile), Prometheus, Grafana

Cara pakai:

cd terraform
terraform init          # Download provider Docker
terraform validate      # Validasi sintaks
terraform plan          # Review perubahan
terraform apply         # Apply infrastructure
terraform destroy       # Hapus semua resources

Blue-Green Deployment

Strategi: Blue (active) -> Green (preview) -> Switch -> Rollback kapan saja.

# Switch ke green
kubectl patch service task-api-blue -p '{"spec":{"selector":{"app":"task-api-preview"}}}'
# Rollback ke blue
kubectl patch service task-api-blue -p '{"spec":{"selector":{"app":"task-api"}}}'
